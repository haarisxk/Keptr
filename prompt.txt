# 17. SECURITY ARCHITECTURE MANDATE

Security is the foundation of Keptr.

Every system, module, service, process, file format, storage layer, synchronization mechanism, user interaction, and architectural decision must be designed under a security-first model.

No component is exempt.

* Design
* Architecture
* Implementation

* Documentation
* Deployment
* Updates
* Maintenance

# 18. THREAT MODEL PHILOSOPHY

Keptr must assume that attackers may attempt to compromise:

* User devices
* Memory
* Storage
* Network traffic
* Cloud infrastructure
* Browser environments
* Extensions
* Update channels
* Backup files
* Synchronization services

The system must be designed to remain secure even when individual layers fail.

Defense in depth is mandatory.

No single point of failure should expose the entire vault.

---

# 19. ZERO-KNOWLEDGE ARCHITECTURE

Keptr shall operate under a strict zero-knowledge security model.

---

## Core Rules

All encryption and decryption operations occur exclusively on authorized client devices.

Plaintext data must never be transmitted to any server.

Master passwords must never leave the device.

Raw encryption keys must never leave the device.

Recovery secrets must never leave the device in plaintext form.

Servers must never possess sufficient information to decrypt user data.

---

## Server Visibility Restrictions

Servers may store:

* Encrypted vault data
* Encrypted metadata where necessary
* Account identifiers
* Synchronization metadata
* Device registration records
* Encrypted backups

Servers must never store:

* Plaintext vault data
* Plaintext credentials
* Plaintext notes
* Plaintext documents
* Plaintext attachments
* Master passwords
* Encryption keys

---

# 20. END-TO-END ENCRYPTION

All user data must be encrypted before leaving the local device.

Encryption must occur prior to:

* Synchronization
* Sharing
* Backup creation
* Cloud uploads

Decryption must only occur on authorized client devices.

Data must remain encrypted:

* At rest
* In transit
* During synchronization
* During backup storage

---

# 21. CRYPTOGRAPHIC POLICY

All cryptographic implementations must prioritize:

* Security
* Stability
* Auditability
* Long-term support
* Industry acceptance

Experimental cryptography must not be used for production systems.

All cryptographic operations must rely on mature, reviewed implementations.

---

# 22. APPROVED CRYPTOGRAPHIC PRIMITIVES

The following represent the baseline cryptographic requirements.

If stronger, mature, widely accepted alternatives become available and are determined to provide meaningful benefits without introducing instability, migration paths may be considered.

---

## Symmetric Encryption

Approved:

* XChaCha20-Poly1305
* AES-256-GCM

Requirements:

* Authenticated encryption
* Confidentiality
* Integrity protection
* Unique nonce usage
* Secure key management

---

## Key Derivation

Approved:

Argon2id

Requirements:

* Memory-hard configuration
* High resistance to GPU attacks
* Tunable security parameters
* Security-first defaults

Minimum baseline:

* 64 MB memory cost
* 4–8 iterations minimum
* Parallelism tuned appropriately

Future versions may increase these parameters as hardware improves.

---

## Digital Signatures

Approved:

Ed25519

Requirements:

* Fast verification
* Strong security
* Compact signatures
* Reliable implementation

---

## Hashing

Approved:

SHA-512

Primary usage:

* Integrity verification
* HMAC construction
* Internal security mechanisms

Where both SHA-256 and SHA-512 are viable options, prefer SHA-512 when practical and appropriate.

Avoid weaker or legacy alternatives unless compatibility requirements explicitly justify their use.

---

## Post-Quantum Readiness

Approved:

ML-KEM

Examples:

* Kyber-768
* Kyber-1024

Requirements:

* Future-proof architecture
* Hybrid deployment strategies when appropriate
* Migration planning for future cryptographic evolution

---

# 23. CRYPTOGRAPHIC IMPLEMENTATION RULES

All cryptographic operations must:

* Use secure randomness
* Use cryptographically secure generators
* Validate inputs
* Validate outputs
* Prevent nonce reuse
* Prevent key reuse where inappropriate
* Fail securely

---

## Forbidden Practices

Never:

* Invent custom encryption algorithms
* Invent custom hash functions
* Invent custom key derivation schemes
* Reuse nonces improperly
* Store plaintext secrets unnecessarily
* Implement security-critical primitives manually when mature implementations exist

---

# 24. KEY HIERARCHY

Keptr must implement strict key separation.

---

## Hierarchical Structure

Master Key

↓

Vault Encryption Key

↓

Domain-Specific Keys

↓

Per-Item Keys

---

Every layer must maintain cryptographic separation.

Compromise of a lower-level key must not expose higher-level keys.

---

# 25. PER-ITEM ENCRYPTION

Every vault item must be encrypted independently.

Each item must possess:

* Unique cryptographic context
* Unique derivation parameters
* Unique identifiers
* Unique salts

Compromise of one encrypted item must not expose any other item.

---

# 26. SALT STRATEGY

Every encrypted item must receive:

* A unique random salt

Requirements:

* Cryptographically secure generation
* No reuse
* Stored alongside encrypted data

---

# 27. PEPPER STRATEGY

Keptr may implement an application-level pepper strategy.

Requirements:

* Separate from vault data
* Separate from user-controlled secrets
* Protected against trivial extraction
* Managed consistently across supported environments

Pepper usage must never weaken portability or recovery guarantees.

---

# 28. KEY ROTATION

Keptr must support secure key rotation.

---

## Automatic Rotation

Support scheduled rotation where appropriate.

---

## Manual Rotation

Users must be able to initiate key rotation.

---

## Rotation Requirements

When keys are rotated:

* Vault data must be re-encrypted securely
* Integrity must be verified
* Recovery mechanisms must remain functional

---

# 29. SHAMIR SECRET SHARING

Support threshold-based recovery models.

Requirements:

# 30. FORWARD SECRECY

# A1. CLOUD BACKEND ARCHITECTURE (ZERO-KNOWLEDGE SYSTEM DESIGN)

## A1.1 Core Principle

The cloud backend is a **dumb encrypted storage and sync relay system**.

It must NEVER:

* decrypt data
* derive keys
* interpret vault structure
* access plaintext

---

## A1.2 System Architecture

The cloud system is composed of:

### 1. API Gateway Layer

* Handles authentication requests
* Routes encrypted payloads
* Rate limiting + abuse prevention

---

### 2. Sync Service Layer

* Receives encrypted vault updates
* Stores versioned encrypted blobs
* Handles conflict metadata only

---

### 3. Object Storage Layer

* Stores:

  * `.kore` encrypted vault objects
  * `.kaps` encrypted file chunks
  * `.kept` backups

* Acts like a **secure object store (S3-like behavior)**

---

### 4. Metadata Index (Non-sensitive only)

Stores ONLY:

* user ID (hashed)
* device IDs (public identifiers)
* sync timestamps
* version vectors
* encrypted blob pointers

---

## A1.3 Sync Protocol Model

### Event-Based Sync System

All changes are represented as:

* immutable encrypted events

Each event contains:

* event_id
* device_id
* timestamp
* encrypted_payload
* hash chain pointer

---

## A1.4 Conflict Model

Hybrid model:

* Vector clock tracking
* Last-write-wins only as fallback
* Merge hints stored encrypted client-side

---

## A1.5 Cloud Security Guarantee

* No plaintext exists at rest or transit
* TLS + additional client encryption required
* Server is always untrusted

---

# A2. CRYPTOGRAPHIC KEY MANAGEMENT (ENGINEERING LEVEL)

---

## A2.1 Key Hierarchy (Strict Model)

```
Master Password
   ↓ (Argon2id KDF)
Master Key (MK)
   ↓
Vault Key (VK)
   ↓
Item Keys (IK)
   ↓
Data Encryption Keys (DEK per item)
```

---

## A2.2 Device Binding Key

Each device generates:

* Ed25519 device identity keypair
* Device Public Key registered to account

Used for:

* sync authentication
* secure session negotiation

---

## A2.3 Key Storage Rules

* Keys NEVER stored in plaintext disk
* Keys only exist in:

  * locked memory
  * secure enclave (if available)
  * ephemeral runtime session

---

## A2.4 Key Rotation System

Triggers:

* periodic rotation
* manual rotation
* compromise detection

Process:

1. generate new VK
2. re-encrypt all IKs
3. update cloud encrypted state
4. invalidate old session keys

---

## A2.5 Secure Sharing Key Model

* Ephemeral session key derived per share
* Forward secrecy enforced
* Share keys auto-expire

# A3. FILE FORMAT BYTE-LEVEL SPECIFICATION

---

## A3.1 .kore FORMAT (Vault Item Container)

### Header Structure

```
[MAGIC 4B] "KORE"
[VERSION 2B]
[FLAGS 2B]
[ITEM_TYPE 1B]
[ALGO_ID 1B]
[SALT 16B]
[NONCE 24B]
[ENCRYPTED_PAYLOAD VAR]
[HMAC 32B]
```

---

## A3.2 Encryption Model

* XChaCha20-Poly1305 (preferred)
* AES-256-GCM fallback

---

## A3.3 Integrity

* HMAC-SHA-512 over full header + payload
* tamper detection mandatory

---

## A3.4 .kaps FORMAT (Encrypted File Chunks)

### Structure

```
[HEADER]
  MAGIC: "KAPS"
  VERSION
  FILE_ID
  CHUNK_COUNT
  ORIGINAL_TYPE
  FILENAME_HASH

[CHUNKS]
  chunk_1 encrypted
  chunk_2 encrypted
  ...

[INDEX TABLE]
  chunk hashes
  offsets
```

---

## A3.5 Streaming Encryption Rules

* chunk size: 1MB–4MB
* each chunk independently encrypted
* supports partial download/decryption

---

## A3.6 .kept FORMAT (Backup Container)

```
[HEADER]
  MAGIC: "KEPT"
  VERSION
  VAULT_ID
  DEVICE_COUNT

[ENCRYPTED_BUNDLE]
  .kore blobs
  .kaps blobs
  metadata snapshot

[INTEGRITY LAYER]
  SHA-512 tree hash root
```
